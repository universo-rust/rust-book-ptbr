# `Rc<T>`, o Smart Pointer de Contagem de Referências

Na maioria dos casos, a ownership é clara: você sabe exatamente qual variável possui um dado. No entanto, há casos em que um único valor pode ter vários donos. Por exemplo, em estruturas de dados em grafo, várias arestas podem apontar para o mesmo nó, e esse nó é conceitualmente possuído por todas as arestas que apontam para ele. Um nó não deve ser limpo a menos que não tenha arestas apontando para ele e portanto não tenha donos.

Você precisa habilitar múltipla posse explicitamente usando o tipo Rust `Rc<T>`, que é uma abreviação de _reference counting_. O tipo `Rc<T>` rastreia o número de referências a um valor para determinar se o valor ainda está em uso. Se houver zero referências a um valor, o valor pode ser limpo sem que referências se tornem inválidas.

Imagine `Rc<T>` como uma TV na sala de estar. Quando uma pessoa entra para assistir TV, ela a liga. Outras podem entrar na sala e assistir TV. Quando a última pessoa sai da sala, ela desliga a TV porque não está mais sendo usada. Se alguém desligar a TV enquanto outros ainda assistem, haverá protesto dos espectadores restantes!

Usamos o tipo `Rc<T>` quando queremos alocar alguns dados no heap para que várias partes do nosso programa leiam e não podemos determinar em tempo de compilação qual parte terminará de usar os dados por último. Se soubéssemos qual parte terminaria por último, poderíamos simplesmente fazer essa parte a dona dos dados, e as regras normais de ownership impostas em tempo de compilação entrariam em vigor.

Note que `Rc<T>` é apenas para uso em cenários de thread única. Quando discutirmos concorrência no Capítulo 16, cobriremos como fazer contagem de referências em programas multithread.

## Compartilhando Dados

Vamos voltar ao nosso exemplo de cons list na Listagem 15-5. Desta vez, criaremos duas listas que compartilham ownership de uma terceira lista. Conceitualmente, isso se parece com a Figura 15-3.

![Uma linked list com o rótulo 'a' apontando para três elementos. Uma lista 'b' aponta para um elemento com 3 que aponta para o primeiro elemento de 'a'. Uma lista 'c' aponta para um elemento com 4 que também aponta para o primeiro elemento de 'a'.](https://doc.rust-lang.org/book/img/trpl15-03.svg)

*Figura 15-3: Duas listas, `b` e `c`, compartilhando ownership de uma terceira lista, `a`*

Criaremos a lista `a` que contém `5` e depois `10`. Em seguida, faremos mais duas listas: `b` que começa com `3` e `c` que começa com `4`. Ambas as listas `b` e `c` continuarão na primeira lista `a` contendo `5` e `10`. Em outras palavras, ambas as listas compartilharão a primeira lista contendo `5` e `10`.

Tentar implementar este cenário usando nossa definição de `List` com `Box<T>` não funcionará, como mostrado na Listagem 15-17.

**Arquivo: src/main.rs (Este código não compila!)**

```rust
enum List {
    Cons(i32, Box<List>),
    Nil,
}

use crate::List::{Cons, Nil};

fn main() {
    let a = Cons(5, Box::new(Cons(10, Box::new(Nil))));
    let b = Cons(3, Box::new(a));
    let c = Cons(4, Box::new(a));
}
```

As variantes `Cons` possuem os dados que guardam, então quando criamos a lista `b`, `a` é movido para `b` e `b` possui `a`. Depois, quando tentamos usar `a` novamente ao criar `c`, não temos permissão porque `a` foi movido.

Poderíamos mudar a definição de `Cons` para guardar referências em vez disso, mas então teríamos que especificar parâmetros de lifetime. Ao especificar parâmetros de lifetime, estaríamos especificando que cada elemento na lista viverá pelo menos tanto quanto a lista inteira. Este é o caso para os elementos e listas na Listagem 15-17, mas não em todo cenário.

Em vez disso, mudaremos nossa definição de `List` para usar `Rc<T>` no lugar de `Box<T>`, como mostrado na Listagem 15-18. Cada variante `Cons` agora guardará um valor e um `Rc<T>` apontando para um `List`. Quando criamos `b`, em vez de tomar posse de `a`, clonaremos o `Rc<List>` que `a` guarda, aumentando assim o número de referências de um para dois e permitindo que `a` e `b` compartilhem ownership dos dados naquele `Rc<List>`. Também clonaremos `a` ao criar `c`, aumentando o número de referências de dois para três. Cada vez que chamamos `Rc::clone`, a contagem de referências dos dados dentro do `Rc<List>` aumentará, e os dados não serão limpos a menos que haja zero referências a eles.

**Arquivo: src/main.rs**

```rust
enum List {
    Cons(i32, Rc<List>),
    Nil,
}

use crate::List::{Cons, Nil};
use std::rc::Rc;

fn main() {
    let a = Rc::new(Cons(5, Rc::new(Cons(10, Rc::new(Nil)))));
    let b = Cons(3, Rc::clone(&a));
    let c = Cons(4, Rc::clone(&a));
}
```

<a id="listagem-15-18"></a>

[Listagem 15-18](#listagem-15-18): Uma definição de `List` que usa `Rc<T>`

Precisamos adicionar uma instrução `use` para trazer `Rc<T>` para o escopo porque não está no prelude. Em `main`, criamos a lista contendo `5` e `10` e a armazenamos em um novo `Rc<List>` em `a`. Depois, quando criamos `b` e `c`, chamamos a função `Rc::clone` e passamos uma referência ao `Rc<List>` em `a` como argumento.

Poderíamos ter chamado `a.clone()` em vez de `Rc::clone(&a)`, mas a convenção do Rust é usar `Rc::clone` neste caso. A implementação de `Rc::clone` não faz uma cópia profunda de todos os dados como a maioria das implementações de `clone` fazem. A chamada a `Rc::clone` apenas incrementa a contagem de referências, o que não leva muito tempo. Cópias profundas de dados podem levar muito tempo. Usando `Rc::clone` para contagem de referências, podemos distinguir visualmente entre os tipos de clone de cópia profunda e os tipos de clone que aumentam a contagem de referências. Ao procurar problemas de desempenho no código, só precisamos considerar os clones de cópia profunda e podemos ignorar chamadas a `Rc::clone`.

## Clonar para Aumentar a Contagem de Referências

Vamos mudar nosso exemplo funcional na Listagem 15-18 para podermos ver as contagens de referência mudando conforme criamos e descartamos referências ao `Rc<List>` em `a`.

Na Listagem 15-19, mudaremos `main` para ter um escopo interno em torno da lista `c`; então, podemos ver como a contagem de referências muda quando `c` sai de escopo.

**Arquivo: src/main.rs**

```rust
enum List {
    Cons(i32, Rc<List>),
    Nil,
}

use crate::List::{Cons, Nil};
use std::rc::Rc;

fn main() {
    let a = Rc::new(Cons(5, Rc::new(Cons(10, Rc::new(Nil)))));
    println!("count after creating a = {}", Rc::strong_count(&a));
    let b = Cons(3, Rc::clone(&a));
    println!("count after creating b = {}", Rc::strong_count(&a));
    {
        let c = Cons(4, Rc::clone(&a));
        println!("count after creating c = {}", Rc::strong_count(&a));
    }
    println!("count after c goes out of scope = {}", Rc::strong_count(&a));
}
```

<a id="listagem-15-19"></a>

[Listagem 15-19](#listagem-15-19): Imprimindo a contagem de referências

Em cada ponto do programa em que a contagem de referências muda, imprimimos a contagem de referências, que obtemos chamando a função `Rc::strong_count`. Esta função se chama `strong_count` em vez de `count` porque o tipo `Rc<T>` também tem um `weak_count`; veremos para que `weak_count` é usado em Prevenindo ciclos de referência usando `Weak<T>`.

Este código imprime o seguinte:

```bash
$ cargo run
   Compiling cons-list v0.1.0 (file:///projects/cons-list)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.45s
     Running `target/debug/cons-list`
count after creating a = 1
count after creating b = 2
count after creating c = 3
count after c goes out of scope = 2
```

Podemos ver que o `Rc<List>` em `a` tem uma contagem de referências inicial de 1; depois, cada vez que chamamos `clone`, a contagem aumenta em 1. Quando `c` sai de escopo, a contagem diminui em 1. Não precisamos chamar uma função para diminuir a contagem de referências como precisamos chamar `Rc::clone` para aumentá-la: a implementação da trait `Drop` diminui a contagem de referências automaticamente quando um valor `Rc<T>` sai de escopo.

O que não podemos ver neste exemplo é que, quando `b` e depois `a` saem de escopo no final de `main`, a contagem é 0, e o `Rc<List>` é limpo completamente. Usar `Rc<T>` permite que um único valor tenha vários donos, e a contagem garante que o valor permaneça válido enquanto qualquer um dos donos ainda existir.

Por meio de referências imutáveis, `Rc<T>` permite compartilhar dados entre várias partes do seu programa apenas para leitura. Se `Rc<T>` permitisse ter várias referências mutáveis também, você poderia violar uma das regras de borrowing discutidas no Capítulo 4: múltiplos empréstimos mutáveis no mesmo lugar podem causar data races e inconsistências. Mas poder mutar dados é muito útil! Na próxima seção, discutiremos o padrão de interior mutability e o tipo `RefCell<T>` que você pode usar em conjunto com um `Rc<T>` para trabalhar com essa restrição de imutabilidade.
